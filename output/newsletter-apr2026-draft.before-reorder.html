What engineers across the org are saying — in their own words

Platform Monitoring Engineering & Tools

Perry Taylor

Automated validation and generation pipeline for EDP metric and event grouping files across 9 active server platforms, eliminating multiple classes of errors (broken formulas, superset violations, version drift, stale events) that previously reached customers undetected. Grouping file generation—previously done by hand across hundreds of XML metric events—is now fully automated, including MSR/RAPL groups. NDA/public tier derivation uses a hybrid regeneration approach that reduced group counts by 16–24%, directly speeding EMON collection runs for customers. Retire_latency substitution across 23 events × 3 visibility tiers (previously a manual multi-file script process) now executes in a single command for GNR. A staging workflow enforces safe edit boundaries so official release files are never modified during development, and diff-based audit trails provide per-release changelogs that previously did not exist.

SAI AICF Applied ML

Yuankun Shi

Claude Code helped optimize the FlashAttention backward pass kernel on Intel Xe GPUs, one of the most compute-intensive operations in LLM training. The main challenge was that Intel’s GPU architecture has undocumented vISA inline assembly constraints, and the only reference was Intel’s closed-source IPEX kernel—we reverse-engineered its compiled assembly to understand the optimization patterns. Key optimizations include rewriting softmax and gradient accumulation hotspots in SIMD32 inline assembly, keeping the attention matrix on-chip to eliminate costly global memory round-trips, and fusing multiple math operations into single instructions. On the on-chip data path, this delivered a 38% speedup on small shapes. The work is built on the open-source SYCLTLA framework, reaching near-parity with Intel’s hand-tuned proprietary kernels while remaining portable and maintainable.

Threading Runtimes GAM

Dan Hoeflinger

A comprehensive performance-modeling framework was built for oneDPL, covering ~15 APIs with a three-view analysis: ideal memory bandwidth roofline, predicted performance (accounting for synchronization and computation costs of the actual implementation), and measured actuals. The framework includes C++ benchmarks to gather hardware-specific constants, making models portable across arbitrary hardware. One algorithmic improvement has already been realized for a class of oneDPL algorithms by reasoning with an agent using this model and its output as context. Claude Code agents enabled rapid iteration—reverse-engineering reusable prompts to extend coverage from one API to 15+ in a fraction of the time manual modeling would have required.

Cost Efficiency BKMs

Practical guidance for getting more out of every dollar

As AIDE scales across the org, cost efficiency matters. Below are some observed best practices that can significantly reduce your AI spend without sacrificing productivity.

New: Usage Analysis Tool

The tokengen portal now includes an AI-powered Usage Analyzer tool that reviews your recent agentic usage patterns and generates specific, personalized recommendations to reduce costs.

Choose the Right Model for the Task

The single biggest observed cost driver is running the most expensive model (Opus 4.6) for everything—including routine tasks like file exploration, debugging, and log parsing where a smaller model performs just as well.

• Claude Code: Stay on Sonnet (the default). Switch to Opus via /model only when you need deep architectural reasoning—then switch back.
• Gemini CLI: Use Auto (Gemini 3) mode—it automatically routes between Pro and Flash based on task complexity, keeping costs down without sacrificing quality.

Manage Your Context Window

Long sessions (500+ messages) drive up costs as the model processes your entire conversation history with every response. This is because all previous messages are resent as input tokens each time to maintain conversational continuity—so the longer your session, the more you pay per message. Use /compact or /clear regularly, and start new sessions after completing sub-tasks.

Be Aware of Agentic Loops

Automated compile–debug–fix cycles—whether triggered by Claude Code sub-agents or by multi-agent frameworks like OpenClaw—can generate massive token volumes through persistent, automated calls. Agentic loops should almost never use the largest frontier models. If you are running any form of agent swarm or automated pipeline, ensure it is configured to use Sonnet, Haiku, or Flash—not Opus. The cost difference is dramatic and the quality difference for iterative tasks is minimal.

Lower-Priced Models Now Available

In addition to Claude and Gemini, lower-priced models including GLM, Minimax, and Kimi are now available through Bedrock. A future newsletter will include a deep dive on configuring OpenCode to use these models, as well as locally hosted models via Gaudi3 on IBM Cloud.
 

Steve’s Corner

Observations on AIDE Enablement

Across the industry, 2x–5x AI-driven productivity is becoming the norm. Flagship models are getting closer to "solving" coding. It’s time to reset teams expectations for what our pace of output should look like.

• Start with ideation. Several teams have run Dungeons focused on one question: what high-value work has been stuck — deprioritized, understaffed, or too complex to tackle? Before jumping into prompts, step away from the day-to-day and invest time in brainstorming and pathfinding. The output isn’t a list of tasks to automate — it’s a set of strategic problems worth throwing AI at.
• Aim AI at the right problems. It’s easy to measure AI in lines of code or tickets closed. But the teams I’ve seen get the biggest impact tend to be the ones who identified work that is both strategic and newly feasible, and pointed AI there.
• Document the strategic value. When you fill out the Tokengen survey, try to go beyond what you built and capture why it matters to Intel. The surveys are how we demonstrate impact to our stakeholders — the more they reflect strategic outcomes rather than just time saved, the stronger our case for continued investment.

Security & Compliance

A reminder that while Claude Code is Infosec-approved for ITS (Intel Top Secret) data, it is based on a shared responsibility model.

Cloud-hosted models are secure, and have clear statements on not using/sharing data. However, ALL clients (Claude Code, OpenCode, VS Code, GitHub Copilot, etc.) are constantly changing, and if used improperly, can lead to unwanted data exfiltration. MCP servers, hooks, third-party extensions, and telemetry can all create paths for data to leave organizational boundaries.
